# 法律引用与幻觉规避操作手册

> 基于V34→V36→V39→V40P1→V41→V42全迭代链修复经验提炼，整合封闭宇宙规则、自动化审计脚本、六维幻觉检测体系、法律知识库、交叉验证引擎和证据索引机制。适用于大模型辅助生成民事及劳动争议司法文书的通用场景。

## 一、核心原则体系

### 1.1 封闭宇宙（Closed-Universe）规则

> 源自 `.trae/rules/legals.md` 最高指令

**证据与事实绑定**：大模型生成的每一句法律事实，必须完全源自 `evidence_manifest.md`（证据索引清单）中所列出并存在于工作区中的真实文件。绝对禁止任何常识性推导、主观脑补或艺术加工。

**禁止盲写条文**：严禁凭记忆引用法条、案例号。所有引用的法律法规必须匹配工作区中的法律法规文件夹，或通过网络检索自权威网站中的真实数据。

**诉审一致原则**：判决主文所支持的金额和项目，必须严格限制在起诉状、上诉状、变更诉求申请书等原始文件界定的范畴内。绝对禁止超诉请裁判。

**中立与公平**：评价用词必须保持司法审判的机械性、严谨性，禁止使用"极其恶劣""明目张胆"等主观带有强烈感情色彩的修辞。

**法律原则底线**：任何裁判结果不得违反基本法律原则，包括但不限于：
- 任何人不得从违法行为中获利原则
- 诚实信用原则
- 公平原则
- 比例原则

### 1.2 零幻觉底线

无法验证的内容，宁可标注"待验证"或删除，绝不编造。宁可文书简短，不可凭空增补。

### 1.3 强制文书结构协议

为了让自动化审计底座（`verify_agent.py`）能够无误解析，生成的判决书草稿 `output/judgment_draft.md` **必须严格包含且仅包含**以下四个标准 Markdown 一级标题：

```
# 一、当事人的诉讼请求与主张
# 二、本院查明事实
# 三、本院认为（说理部分）
# 四、判决如下（判决主文）
```

各部分内容要求：
- **第一部分**：完整、不遗漏地列出原告/上诉人的具体诉请项目及金额上限
- **第二部分**：每一行事实确认，句末必须强制括号标注来源，格式如：（见《文件名.md》）
- **第三部分**：每一项支持或驳回结论，必须在同一段落内同时包含适用的法律大前提《XX法》第X条和支撑的事实小前提见《XX证据.md》
- **第四部分**：金钱项的最终裁决，任意一项金额和项目，绝对不得超出第一部分或诉求变更申请书中的额度上限

## 二、六维幻觉分类检测体系（H-1至H-6）

> 源自V39幻觉检测报告，经V40P1→V41→V42迭代验证

### H-1：无源编造事实（最高风险）

**定义**：文书中出现的任何事实陈述，无任何证据文件支撑，系大模型凭空生成。

| 子类型 | 特征 | 历史案例 | 防范措施 |
|:---|:---|:---|:---|
| 法条虚构 | 编造不存在的司法解释文号或法条内容 | V40P1引用"法释〔2025〕12号"（当时尚未公布） | 所有司法解释文号必须联网验证；法条引用必须逐字核对原文 |
| 事实编造 | 编造不存在的事实情节 | V36编造"肝功能异常""本院于2026年1月30日裁定中止审理" | 每项事实必须有证据引注；无证据则标注"上诉人主张...，但截至本操作时未见相关书证支持" |
| 内部版本引用 | 判决书正文中出现对AI生成文档版本的自我引用 | V39正文中出现"V38版本""V38此前""本版本" | 全部删除对内部版本号的引用；版本说明仅保留于文书末尾AI生成声明区域 |
| 身份细节编造 | 编造当事人或第三人的具体身份信息 | V39编造王永鑫"2017年入职湖北省药品监督管理局" | 细节无法从证据材料直接核实时，改为更笼统的表述 |
| 金额编造 | 编造未经证据支撑的具体金额 | V39编造双创人才资助"200万元""100万元" | 核实证据中是否明确记载此金额；若无直接记载，删除具体数字 |
| 事实经过杜撰 | 编造案件事实经过的细节 | 编造"双方于2024年3月协商解除"等未经证据证实的事实 | 每项事实经过必须有证据引注；无证据则标注"上诉人主张...，但截至本操作时未见相关书证支持" |
| 判决结果杜撰 | 编造类案的判决结果或裁判宗旨 | 编造"某案判决支持了二倍工资差额"等不存在的裁判结果 | 类案判决结果必须联网验证；不可验证的必须标注"待验证" |
| 类案案号杜撰 | 无中生有地编造看似合理的案例编号 | 编造"(2024)苏01民终1234号"等不存在的案例 | 类案案号必须联网验证；不可验证的必须标注"待验证" |
| 证据内容杜撰 | 对证据文件内容进行编造或歪曲 | 将"股东会议决议"编造为"审计报告" | 引用证据内容必须与证据文件原文逐字核对 |

**检测方法**：
1. 提取文书中所有事实陈述
2. 逐项核对 `evidence_manifest.md` 中是否有对应证据文件
3. 无对应证据的事实标记为H-1幻觉
4. 交叉验证引擎核对事实陈述与原始证据文件内容

### H-2：法律适用错误（高风险）

**定义**：引用的法条不存在、已废止、内容错误或与案件事实不涵摄。

| 子类型 | 特征 | 历史案例 | 防范措施 |
|:---|:---|:---|:---|
| 法条不存在 | 引用的法律/司法解释文号在权威渠道无法查到 | V40P1引用"法释〔2025〕12号" | 联网检索国家法律法规数据库确认 |
| 法条已废止 | 引用的法条已被新法取代 | 引用《合同法》而非《民法典》 | 确认法条效力状态 |
| 法条内容错误 | 引用的条文内容与原文不一致 | 大模型凭记忆概括法条 | 逐字核对条文原文 |
| 拟议中法条作为生效法律引用 | 将尚未生效的司法解释草案作为裁判依据 | V40P1将拟议中的法释〔2025〕12号作为已生效法律引用 | 确认法条生效日期；标注"尚未生效"或删除 |
| 法律方法论替换 | 将起诉状中的法律依据替换为另一套法律体系 | V39将劳部发〔1995〕223号替换为《民法典》违约责任 | 若变更法律依据，必须在说理部分论证变更理由 |
| 法条张冠李戴 | 将A法的条款错误应用于B法的事实场景 | 将《劳动合同法》第82条（二倍工资）误用于违法解除赔偿金场景 | 核实法条适用场景与案件事实是否匹配；劳动法特别规定优先于民法一般规定 |
| 类案杜撰 | 无中生有地编造类案案号、裁判宗旨或判决结果 | 编造"(2024)苏01民终1234号"等不存在的案例 | 类案案号必须联网验证；不可验证的必须标注"待验证" |
| 程序时效杜撰 | 对法定的程序时效不理解，杜撰程序性日期 | 9月9日一审送达，却根据上诉状修改日期认定上诉时11月（明显超出15日上诉期） | 严格按法定时效计算关键日期；上诉期15日、仲裁时效1年、再审申请期6个月等 |
| 仲裁时效张冠李戴 | 将民事时效规则错误适用于劳动争议仲裁 | 将3年民事诉讼时效用于劳动争议1年仲裁时效 | 劳动争议适用《劳动争议调解仲裁法》第27条1年时效，不适用《民法典》3年时效 |
| 法律法规变更未追踪 | 引用已修订但未标注修订版本的法条 | 引用旧版《公司法》条文而未标注2023修订 | 建立法律法规变更追踪机制，定期更新本地法条库 |

**检测方法**：
1. 提取文书中所有法条引用
2. 逐条联网验证法条存在性、效力状态、内容准确性
3. 与起诉状/上诉状中的法律依据对比，识别方法论替换
4. 检测法条张冠李戴：法条引用与案件事实场景是否匹配
5. 检测类案杜撰：类案案号是否可验证，裁判宗旨是否与公开裁判文书一致
6. 检测程序时效杜撰：关键程序性日期是否与法定时效一致
7. 检测仲裁时效张冠李戴：是否将民事3年时效错误适用于劳动争议1年仲裁时效
8. 法律知识库验证：利用本地法条库对法条引用进行自动化校验

### H-3：证据链断裂（高风险）

**定义**：证据→事实→法律→结论的推理链条存在缺失环节。

| 子类型 | 特征 | 防范措施 |
|:---|:---|:---|
| 证据→事实断裂 | 事实认定无证据支撑 | 每项事实必须有证据引注 |
| 事实→法律断裂 | 事实与法条不涵摄 | 确认案件事实满足法条的全部构成要件 |
| 法律→结论断裂 | 法条规定的法律效果与裁判主文不一致 | 确认法条效果与判决主文对应 |
| 三段论缺失大前提 | 说理部分仅有事实和结论，缺少法律依据 | `verify_agent.py` 自动检测 |
| 三段论缺失小前提 | 说理部分仅有法律和结论，缺少证据锚点 | `verify_agent.py` 自动检测 |

**检测方法**：`verify_agent.py` 的 `verify_syllogism_complete_chain()` 方法自动扫描说理部分，检测三段论完整性。

### H-4：主观臆断/修辞过度（中风险）

**定义**：使用主观评价性语言替代客观事实描述，或对当事人主观状态进行无证据推断。

| 子类型 | 特征 | 历史案例 | 防范措施 |
|:---|:---|:---|:---|
| 道德评价 | 使用"恶意""卑劣""无耻"等评价性词汇 | V39/V40P1中"恶意遮蔽""恶意减资" | 替换为中性表述 |
| 程序事实主观推断 | 对法庭内部程序事实进行主观定性 | V39"逾期四个多月""迟至" | 标注为待法庭确认事项 |
| 意图推断 | 推断当事人的主观意图 | V40P1"主观意图" | 删除主观修饰，仅描述客观行为 |
| 情感化修辞 | 使用"令人愤慨""触目惊心"等 | V34"欺诈""主观恶意极为严重" | 全部替换为法言法语 |

**保留例外**：法条原文使用"恶意"（如《民法典》第154条"恶意串通"）、司法解释规范表述、法律概念对立统一（善意取得vs恶意取得）。

### H-5：诉求边界突破（高风险）

**定义**：判决主文支持的金额或项目超出起诉状/上诉状/变更诉求申请书的范围。

| 子类型 | 特征 | 历史案例 | 防范措施 |
|:---|:---|:---|:---|
| 金额冒顶 | 判决支持金额超过诉请上限 | V39待岗工资1,488,248元远超起诉状357,000元 | `verify_agent.py` 自动检测 |
| 项目越权 | 判决支持了未曾主张的项目 | V39"正常履职期奖金759,000元"在起诉状中无对应项目 | 判决项目必须与诉请项目一一对应 |
| 计算方法替换 | 以不同方法论重新计算导致金额超出 | V39以同工同酬50,000元/月替换起诉状17,000元/月基准 | 若变更计算方法，必须论证变更理由 |
| 法律依据替换 | 将起诉状中的法律依据替换为另一套体系 | V39将劳部发〔1995〕223号替换为《民法典》违约责任 | 若变更法律依据，必须论证变更理由 |
| 计算错误 | 加班工资、二倍工资差额等关键计算项的公式或数值错误 | 将150%加班费率误算为200%，或月工资基数取错 | 所有涉及金额的计算项均需独立核验 |
| 逻辑循环 | 以推定性金额为基数计算惩罚性赔偿 | 以"二倍工资差额"为基数计算"加付赔偿金" | 区分补偿性金额与惩罚性金额的基数 |
| 数字引用不一致 | 同一金额在不同段落出现不同数值 | 事实部分认定"拖欠工资50,000元"，判决主文却写"55,000元" | 全文交叉校验同一事项的金额引用 |
| 利息基数错误 | 以惩罚性金额为基数计算利息 | 以二倍工资差额为基数计算利息 | 利息基数应为实际损失金额，不含惩罚性赔偿 |

**检测方法**：`verify_agent.py` 的 `verify_strict_scope_containment()` 方法自动提取诉请金额上限，与判决主文逐项比对。计算核验模块对关键计算项进行公式还原和数值校验。

### H-6：非文本证据穿透失败（检测端能力边界）

**定义**：大模型无法直接验证非文本证据（录音、视频、实物证据）中的事实，可能导致误判。

| 特征 | 防范措施 |
|:---|:---|
| 录音/视频内容无法验证 | 标注"据当事人陈述/据录音整理文字"，不作为独立定案依据 |
| 实物证据无法检验 | 仅引用证据清单中的描述，不编造检验细节 |
| 庭审记录无法获取 | 标注"据当事人回忆/据庭审笔录（待确认）" |

### 3.4 交叉验证引擎

#### 验证流程

```
1. 提取判决书中的事实陈述、法条引用、案例引用
2. 与证据材料进行原始文件核对
3. 与法律知识库中的法条原文进行比对
4. 与案例库中的案例信息进行比对
5. 标记匹配/不匹配/未找到的验证结果
6. 对不匹配项分析可能原因
7. 给出具体、可执行的修正建议
```

#### 验证类型

| 验证类型 | 说明 | 输出 |
|:---|:---|:---|
| 证据引注验证 | 核对引注是否存在于证据清单 | 匹配/不匹配 |

## 四、法律引用校验流程（十步法）

> 整合V41六步法与V39方法论替换检测，V42新增法律知识库验证步骤

### 步骤1：法条存在性验证

| 操作 | 说明 |
|:---|:---|
| 联网检索确认 | 每一条引用的法律/司法解释，必须通过权威渠道确认其真实存在 |
| 权威渠道优先级 | ①国家法律法规数据库 ②最高人民法院官网 ③北大法宝 ④裁判文书网 |
| 文号核对 | 司法解释文号格式：法释〔年份〕序号。核对年份与序号是否对应真实发布记录 |
| 生效日期核对 | 确认法条在案件裁判时点是否已生效 |

### 步骤2：条文内容逐字核对

| 操作 | 说明 |
|:---|:---|
| 全文引用核对 | 引用法条时，逐字核对条文原文，不得凭记忆概括 |
| 条款项号核对 | 确认引用的条、款、项编号与原文一致 |
| 但书/除外条款 | 引用法条时必须完整引用，不得遗漏但书或除外条款 |
| 本地法条库核对 | 优先核对法律知识库中的法条原文 |

### 步骤3：法条效力状态确认

| 操作 | 说明 |
|:---|:---|
| 是否废止 | 确认法条是否已被废止或修订 |
| 是否被新法取代 | 如《民法典》生效后，《合同法》《侵权责任法》等已废止 |
| 过渡期适用 | 确认法条的时间效力范围是否覆盖案件事实发生期间 |
| 司法解释时效 | 司法解释是否已被新的司法解释取代 |

### 步骤4：法条与案件事实的涵摄关系

| 操作 | 说明 |
|:---|:---|
| 构成要件匹配 | 确认案件事实满足法条的全部构成要件 |
| 法律效果对应 | 确认法条规定的法律效果与裁判主文一致 |
| 特别法优先 | 劳动争议优先适用《劳动合同法》《劳动法》 |
| 一般法补充 | 特别法无规定时方可适用一般法（如《民法典》） |

### 步骤5：法律方法论一致性检查

| 操作 | 说明 |
|:---|:---|
| 与起诉状法律依据对比 | 检查判决书采用的法律方法论是否与起诉状/上诉状一致 |
| 方法论变更论证 | 若变更法律依据，必须在说理部分论证变更理由 |
| 避免重复评价 | 同时适用加付赔偿金和违约损害赔偿金时，需论证二者不构成重复评价 |

### 步骤6：类案引用诚实标注

| 操作 | 说明 |
|:---|:---|
| 可验证案例 | 最高人民法院指导性案例、公报案例可通过官方渠道验证 |
| 不可验证案例 | 无法通过公开渠道验证案号的类案，必须标注"模拟类案参考"或"待验证" |
| 禁止编造案号 | 不得编造看似合理的案号格式 |
| 裁判要旨核对 | 类案裁判要旨必须源于裁判文书原文，不得概括编造 |

### 步骤7：引用格式规范化

| 要素 | 格式要求 |
|:---|:---|
| 法律名称 | 全称+简称首次标注 |
| 条款引用 | 第X条第X款第X项，层次分明 |
| 司法解释 | 全称+文号 |
| 行政规章 | 全称+文号 |

### 步骤8：内部版本引用清除

| 操作 | 说明 |
|:---|:---|
| 扫描版本号引用 | 检查文书中是否出现"VXX版本""本版本""此前版本"等 |
| 全部删除 | 判决书正文中不得出现任何对AI生成文档版本的引用 |
| 版本说明保留位置 | 版本迭代说明仅保留于文书末尾的AI生成声明区域 |

### 步骤9：法律知识库验证

> V42新增步骤

| 操作 | 说明 |
|:---|:---|
| 本地库比对 | 将法条引用与法律知识库中的法条原文进行自动比对 |
| 效力状态检查 | 自动检查法条是否现行有效、是否已修订或废止 |
| 适用场景匹配 | 自动检测法条引用与案件事实场景是否匹配（张冠李戴检测） |

### 步骤10：附录法律法规清单

| 操作 | 说明 |

## 六、诉请边界审计流程

> 源自 `verify_agent.py` 的 `verify_strict_scope_containment()` 方法

### 6.1 诉请金额上限提取

`verify_agent.py` 通过三层正则引擎从诉状文本中提取各项诉请的金钱最高上限：

| 层级 | 匹配模式 | 说明 |
|:---|:---|:---|
| 第一层 | 序号开头的诉请项+金额 | 如"1、判令被告支付拖欠工资50,000元" |
| 第二层 | 关键词+金额 | 如"加班工资共计143,356.32元" |
| 第三层 | 兜底模式 | 如"拖欠工资XXX元""二倍工资XXX元" |

### 6.2 判决主文金额扫描

脚本自动提取"四、判决如下"部分中的金钱项支持额，与诉请上限逐项比对：

| 检测项 | 错误类型 | 说明 |
|:---|:---|:---|
| 项目越权 | 越权裁判幻觉 | 判决支持了未曾主张的项目 |
| 金额冒顶 | 超诉请裁判幻觉 | 判决支持金额超过诉请上限 |

### 6.3 诉审一致原则操作规范

| 规则 | 说明 |
|:---|:---|
| 项目对应 | 判决支持的每个项目必须在起诉状/上诉状中有对应诉请 |
| 金额上限 | 判决支持的金额必须 ≤ 诉请中该项目的最高金额 |
| 变更诉求 | 若有变更诉求申请书，以变更后的金额为准 |
| 新增项目 | 二审新增的诉请项目需经调解程序，调解不成则另行起诉 |

### 6.4 运行审计脚本

```bash
python scripts/verify_agent.py
```

- 返回 `[AUDIT_PASSED]`：文书未超出任何诉请上界
- 返回 `[AUDIT_FAILED]`：触发高阶硬性熔断，需根据报告修正后重新运行

## 七、主观修辞清除流程（四步法）

### 步骤1：全文扫描识别

| 扫描关键词 | 风险等级 | 说明 |
|:---|:---|:---|
| 恶意、卑劣、无耻、明目张胆 | 高 | 道德评价类 |
| 令人愤慨、触目惊心、令人不齿 | 高 | 情感色彩类 |
| 极其、严重、公然、肆无忌惮 | 中 | 强度修饰类 |
| 主观意图、蓄意、别有用心 | 高 | 意图推断类 |
| 逾期、迟至、无故拖延 | 中 | 程序事实主观推断类 |
| 欺诈、主观恶意极为严重 | 高 | 过度定性类 |

### 步骤2：分类处理

| 类别 | 处理方式 | 示例 |
|:---|:---|:---|
| 纯主观评价 | 替换为中性表述 | "恶意遮蔽"→"不当遮蔽" |
| 程序事实推断 | 标注待确认 | "逾期四个多月"→客观描述 |
| 意图推断 | 删除主观修饰或加"嫌疑" | "恶意串通取证"→"串通取证的嫌疑" |
| 客观效果描述 | 从主观定性转为客观描述 | "逃废债务的积极姿态"→"客观上形成了逃避债务的效果" |

### 步骤3：法律必要用法保留

| 保留情形 | 示例 | 保留理由 |
|:---|:---|:---|
| 法条原文术语 | 恶意串通（民法典第154条） | 法条原文使用"恶意" |
| 司法解释规范表述 | 恶意隐匿证据 | 司法解释中的规范表述 |
| 法律概念对立统一 | 善意取得vs恶意取得 | 法律概念的对立统一 |

### 步骤4：残留扫描验证

全文再次搜索所有修辞关键词，确认无遗漏。

## 九、自动化审计脚本使用指南

| `vector_index.py` | 向量索引模块 | 法律文本矢量化和检索 |
| `report_builder.py` | 报告生成器 | 结构化检测报告 |
| `workflow_orchestrator.py` | 工作流编排模块 | 多子代理并行检测协调器，借鉴claude-code的coordinator模式 |

### 9.3 运行流程

```
1. 装载 evidence_manifest.md → 构建有效证据文件名集合
2. 读取诉状/上诉状/变更申请 → 提取诉请金额上限
3. 读取判决书 → 执行六维幻觉检测
4. 法律知识库验证 → 法条引用与原文比对
5. 交叉验证 → 事实陈述与原始证据核对
6. 诉请边界审计 → 逐项比对判决金额与诉请上限
7. 三段论审计 → 检查引注防伪+说理完整性
8. 生成检测报告 → 输出结构化报告
```

### 9.4 工作流编排模式

> V42新增：借鉴claude-code的coordinator模式，实现多子代理并行检测

#### 9.4.1 设计原理

单Agent处理超长上下文时，LLM注意力和返回质量会显著下降。工作流编排模式将幻觉检测任务分解为多个独立子任务，由多个子代理并行执行，每个子代理仅处理一个阶段的检测，大幅减少单次LLM调用的上下文长度。

#### 9.4.2 工作流阶段

| 阶段 | 名称 | 检测维度 | 优先级 | 输入章节 |
|:---|:---|:---|:---|:---|
| 1 | 文书结构检测 | 强制文书结构协议 | 关键 | 全文 |
| 2 | 证据引注与事实来源检测 | H-1 | 关键 | 查明事实+说理 |
| 3 | 法条引用与法律适用检测 | H-2 | 高 | 说理+附录 |
| 4 | 三段论完整性检测 | H-3 | 高 | 说理 |
| 5 | 主观修辞检测 | H-4 | 中 | 全文 |
| 6 | 诉求边界检测 | H-5 | 关键 | 诉请+判决主文 |
| 7 | 交叉验证与原始文件核对 | H-1/H-2/H-3 | 高 | 查明事实+证据 |
| 8 | 程序时效检测 | H-2 | 高 | 全文 |

#### 9.4.3 令牌预算约束

| 参数 | 默认值 | 说明 |
|:---|:---|:---|
| max_context_tokens | 128000 | LLM最大上下文令牌数 |
| RESERVED_SYSTEM_TOKENS | 4000 | 系统提示词预留 |
| RESERVED_OUTPUT_TOKENS | 8000 | 输出预留 |
| 可用输入令牌 | 116000 | max - system - output |

每个子任务的输入文本按令牌预算截断，确保不超出LLM处理能力。

#### 9.4.4 MCP工作流工具

| 工具名 | 功能 | 说明 |
|:---|:---|:---|
| `create_detection_workflow` | 创建并行检测工作流 | 返回工作流ID和并行任务列表 |
| `get_workflow_tasks` | 获取可并行执行的子任务 | 返回子任务列表和提示词 |
| `update_workflow_task` | 更新子任务状态 | 完成后调用 |
| `get_workflow_status` | 获取工作流状态 | 查看进度 |
| `aggregate_workflow_results` | 汇总检测结果 | 生成综合评估 |
| `list_workflows` | 列出所有工作流 | 概览 |

#### 9.4.5 工作流执行流程

```
1. 调用 create_detection_workflow → 获取工作流ID和8个并行任务
2. 对每个任务，Agent使用自身LLM并行执行检测
3. 每个任务完成后，调用 update_workflow_task 更新状态
4. 所有任务完成后，调用 aggregate_workflow_results 汇总结果
5. 基于汇总结果生成最终检测报告
```

## 十六、幻觉修改建议指南

### 16.1 类案案号杜撰

**诊断**：判决书引用了无法验证的类案案号，大模型可能杜撰了格式正确但实际不存在的案号。

**人工修改步骤**：
1. 在中国裁判文书网(https://wenshu.court.gov.cn)搜索该案号，确认是否存在
2. 若案号不存在：删除该类案引用，或替换为可验证的真实类案
3. 若案号存在但内容不符：修正类案的裁判宗旨描述，确保与原文一致
4. 在evidence_manifest.md的"类案类"分区添加已验证的类案判决书文件路径
5. 重新运行：`python update_evidence_manifest.py --add '类案文件路径'`
6. 重新扫描验证：`python multi_version_scan.py`

**LLM提示词修正**：在生成判决书时，使用"模拟类案A-01"等标识替代真实案号格式，并添加"模拟类案参考"标注。

### 16.2 引注欺诈

**诊断**：判决书引用的证据名称与证据索引清单中的文件名不匹配。

**人工修改步骤**：
1. 对比判决书中的证据引用名与evidence_manifest.md中的文件名
2. 确认证据30的实际文件名：证据30-1(股东会议决议对账单等.pdf) + 证据30-2(对账单部分.pdf)
3. 修改判决书中的证据引用，使用与清单一致的名称
4. 若清单名称有误：运行 `python update_evidence_manifest.py --update '证据30' --desc '正确描述'`
5. 重新运行幻觉检测确认引注匹配

**LLM提示词修正**：在生成判决书时，从evidence_manifest.md中提取精确的证据文件名，禁止自行编造或简化证据名称。

### 16.3 程序日期杜撰

**诊断**：判决书中的程序性日期（立案日期、开庭日期、判决日期等）可能被大模型编造。

**人工修改步骤**：
1. 核实案件真实时间线：查阅一审裁定书、仲裁裁决书等程序性文书
2. 将判决书中所有日期与真实时间线逐一比对
3. 修正不一致的日期，确保与程序性文书完全一致
4. 在evidence_manifest.md中添加程序性文书（如裁定书、庭审笔录等）
5. 重新扫描验证

**LLM提示词修正**：在生成判决书时，从起诉状、仲裁裁决书等文件中提取真实日期，禁止凭空编造程序性日期。

### 16.4 引用格式问题

**诊断**：法条引用格式不规范，缺少款/项序号或格式不一致。

**人工修改步骤**：
1. 查阅法律法规原文，确认引用的法条是否存在且序号正确
2. 补充完整的款/项序号，如"第三十五条第一款"而非仅"第三十五条"
3. 将法律法规原文添加到evidence_manifest.md的"法律类"分区
4. 运行 `python update_evidence_manifest.py --from-report 检测报告路径`
5. 重新扫描验证

**LLM提示词修正**：在生成判决书时，从法律法规原文中提取精确的条文序号，包含款/项级别。

### 16.5 项目越权

**诊断**：判决主文支持的金额或项目超出了原告/上诉人的诉请范围。

**人工修改步骤**：
1. 仔细核对起诉状和变更诉求申请书中的每一项诉请及金额上限
2. 将判决主文的每一项与对应诉请逐一比对
3. 删除超出诉请范围的判决项，或将金额调整至诉请上限以内
4. 若诉请本身有误：先修正起诉状，再重新生成判决书
5. 重新运行幻觉检测确认无超诉请裁判

**LLM提示词修正**：在生成判决主文时，严格以起诉状和变更诉求申请书为边界，任何金额不得超过诉请上限。

### 16.6 参照系替换

**诊断**：判决书在计算赔偿时替换了参照系（如用同工同酬替代原告历史收入），但未说明理由。

**人工修改步骤**：
1. 确认原告诉请中使用的计算参照系
2. 若需替换参照系：在说理部分充分论证替换的必要性和法律依据
3. 引用支持参照系替换的类案或法条
4. 确保替换后的计算结果不超出诉请范围
5. 重新扫描验证

**LLM提示词修正**：在计算赔偿金时，优先使用原告自身历史收入作为参照系；如需替换，必须在说理部分论证并引用法律依据。

### 16.7 非文本证据未标注

**诊断**：判决书引用了录音、视频等非文本证据，但未标注证据的来源形式和整理方式。

**人工修改步骤**：
1. 在证据引用处添加来源形式标注，如"（见《证据5.1 通话录音文字整理》）"
2. 确保非文本证据的文字整理版本已添加到evidence_manifest.md
3. 若无文字整理版：制作文字整理稿并添加到证据清单
4. 重新扫描验证

**LLM提示词修正**：引用非文本证据时，必须同时引用其文字整理版本，并标注"据XX整理"或"据XX笔录"。

### 16.8 三段论缺失-法律依据

**诊断**：说理部分的结论缺少法律依据（大前提），无法形成完整的法律推理三段论。

**人工修改步骤**：
1. 找到说理部分中缺少法律依据的结论句
2. 查阅相关法律法规，找到支撑该结论的具体法条
3. 在结论前添加法律依据引用，格式："根据《XX法》第X条，..."
4. 确保引用的法条与结论逻辑一致
5. 重新扫描验证三段论完整性

**LLM提示词修正**：每个裁判结论必须同时包含法律依据（大前提）和事实证据（小前提），形成完整三段论。

### 16.9 三段论缺失-证据引用

**诊断**：说理部分的结论缺少证据引用（小前提），事实认定未绑定证据来源。

**人工修改步骤**：
1. 找到说理部分中缺少证据引用的事实认定
2. 在evidence_manifest.md中查找对应的证据文件
3. 在事实认定后添加证据来源标注，格式："（见《证据XX_文件名》）"
4. 若无对应证据：该事实认定应标注为"上诉人主张...但未见相关书证支持"
5. 重新扫描验证

**LLM提示词修正**：每个事实认定必须绑定证据来源，无证据支撑的事实应如实标注为"未见书证支持"。

### 16.10 模拟类案已标注

**诊断**：类案已标注"模拟"属性，但案号格式仍可能误导读者以为是真实案例。

**人工修改步骤**：
1. 将模拟类案的案号格式改为"模拟类案A-01"等非真实案号格式
2. 在引用处添加醒目的"（模拟参考，非真实案例）"标注
3. 在判决书末尾添加免责声明，说明模拟类案仅供参考

**LLM提示词修正**：模拟类案应使用"模拟类案A-01"格式，避免使用真实案号格式。

## 十七、证据清单管理

```bash
python update_evidence_manifest.py --check
```

python update_evidence_manifest.py --add "C:\path\to\新证据.md"

python update_evidence_manifest.py --update "证据30" --desc "正确描述"
