## Overview

This tool is an MCP (Model Context Protocol) server designed to detect hallucination issues in judicial documents generated by large language models.

**Core Design Philosophy**: The server itself does not call any LLM API. It only provides a rule engine and prompt templates, allowing AI Agents (such as Trae, Claude Desktop, etc.) to use their own LLM for semantic-level judgments.

**Closed-Universe Rule**: Every legal fact generated by the LLM must be fully sourced from real files listed in `evidence_manifest.md` (the evidence index) and present in the workspace. Common-sense inference, subjective fabrication, or artistic embellishment are absolutely prohibited.

**Mandatory Document Structure**: Generated judgment drafts must contain exactly four standard Markdown level-1 headings:

```
# 一、当事人的诉讼请求与主张
# 二、本院查明事实
# 三、本院认为（说理部分）
# 四、判决如下（判决主文）
```

## Quick Start

Get started in 60 seconds:

```bash
# 1. Clone the repository
git clone https://github.com/your-username/legal-hallucination-mcp.git
cd legal-hallucination-mcp

# 2. Install
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
pip install -e .

# 3. Copy environment template
cp .env.example .env

# 4. Run detection on sample files
python scripts/verify_agent.py \
  --vault-root ./vault_mirror \
  --manifest ./vault_mirror/.trae/evidence_manifest.md \
  --draft ./vault_mirror/output/sample_judgment.md
```

`[AUDIT_PASSED]` means the document is clean; `[AUDIT_FAILED]` lists specific issues.

## Six-Dimension Hallucination Detection Framework

| Dimension | Code | Detection Scope | Risk Level |
|:---|:---|:---|:---|
| Sourceless Fabrication | H-1 | Citation fraud, internal version references, fabricated legal citations, unbound fact sources, fabricated case numbers | Highest |
| Law Misapplication | H-2 | Repealed laws, improper citation format, legal methodology replacement, fabricated procedural dates | High |
| Evidence Chain Break | H-3 | Missing major/minor premises in syllogism, missing document structure | High |
| Subjective Rhetoric | H-4 | Moral judgments, intent inference, emotional rhetoric | Medium |
| Claim Boundary Breach | H-5 | Judgment amounts exceeding claim limits, unauthorized items, calculation errors, interest base errors | High |
| Non-text Evidence Failure | H-6 | Unmarked audio/video/forensic evidence sources | Low |

Skill templates for each dimension are located at `src/legal_hallucination_mcp/skills/dimensions/`:

```
h1_sourceless_fabrication.md   # Sourceless Fabrication
h2_law_misapplication.md       # Law Misapplication
h3_evidence_chain_break.md     # Evidence Chain Break
h4_subjective_rhetoric.md      # Subjective Rhetoric
h5_claim_boundary_breach.md    # Claim Boundary Breach
h6_nontext_evidence_fail.md    # Non-text Evidence Failure
```

## Architecture

```
┌─────────────┐     MCP Protocol    ┌──────────────────────┐
│  AI Agent   │ ◄──────────────────►│  MCP Server          │
│ (Trae/Claude)│                     │                      │
│             │                      │  ┌────────────────┐  │
│  Own LLM    │◄──Prompt Template───│  │ Rule Engine     │  │
│  Semantic   │───LLM Response─────►│  │ (Regex+Numeric) │  │
│  Judgment   │                      │  └────────────────┘  │
│             │                      │  ┌────────────────┐  │
│             │                      │  │ Evidence Index  │  │
│             │                      │  └────────────────┘  │
│             │                      │  ┌────────────────┐  │
│             │                      │  │ Claim Parser    │  │
│             │                      │  └────────────────┘  │
│             │                      │  ┌────────────────┐  │
│             │                      │  │ Law Citation    │  │
│             │                      │  │ Checker         │  │
│             │                      │  └────────────────┘  │
│             │                      │  ┌────────────────┐  │
│             │                      │  │ Report Builder  │  │
│             │                      │  └────────────────┘  │
│             │                      │  ┌────────────────┐  │
│             │                      │  │ Audit Harness   │  │
│             │                      │  │ (verify_agent)  │  │
│             │                      │  └────────────────┘  │
└─────────────┘                      └──────────────────────┘
```

**Rule Engine**: Deterministic detection based on regular expressions, pattern matching, and numeric comparison. No LLM required — returns results instantly.

**Audit Harness**: `scripts/verify_agent.py` implements claim boundary auditing and syllogism completeness auditing, outputting `[AUDIT_PASSED]` or `[AUDIT_FAILED]`.

**Prompt Templates**: For semantic judgments beyond the rule engine's scope (e.g., fact fabrication, legal text verification), the server returns prompt templates for the Agent to send to its own LLM.

## Detection Report Structure

Generated reports contain the following sections:

1. **Risk Overview** — Score, risk grade, dimension summary
2. **Document Structure Check** — Four required section headings
3. **Citation Fraud Detection (H-1)** — Evidence citations vs. evidence index
4. **Fact Source Binding Detection (H-1)** — Fact statements bound to evidence sources
5. **Law Citation Detection (H-2)** — Repealed laws, format issues, misapplied citations
6. **Syllogism Completeness (H-3)** — Major premise (law) + minor premise (evidence)
7. **Subjective Rhetoric Detection (H-4)** — Moral judgments, intent inference, emotional language
8. **Claim Boundary Detection (H-5)** — Judgment amounts vs. claim limits
9. **Non-text Evidence Detection (H-6)** — Audio/video source annotations
10. **Cross-verification & Original Document Check** — Multi-source comparison of facts vs. evidence
11. **Law Knowledge Base Verification** — Citation comparison with law database
12. **Detailed Flags by Dimension** — Original text, location, severity for each flag
13. **Comprehensive Correction Suggestions** — Prioritized suggestions
14. **Urgent Fix Priority** — P0 to P3 graded fix guidance
15. **Post-generation Self-check Checklist** — A-F categories

Report filename convention: `{AgentName}_{LLMNameVersion}_{ContentSummary}_{Version}_{YYYYMMDD}.md`

Reports use GitHub Alerts format (`> [!NOTE]`, `> [!WARNING]`, `> [!TIP]`, `> [!IMPORTANT]`, `> [!CAUTION]`) to highlight different types of annotations, reviews, and suggestions.

## Risk Scoring System

| Grade | Score Range | Meaning |
|:---:|:---|:---|
| A | 0-5 | Minimal risk: virtually no hallucination traces |
| B | 5-15 | Low risk: minor hallucinations, no impact on judgment |
| C | 15-30 | Medium risk: multiple hallucinations, may affect fairness |
| D | 30-50 | High risk: dense hallucinations, judgment credibility questionable |
| F | 50-100 | Critical risk: hallucinations rampant, document fundamentally unreliable |

The scoring system uses diminishing penalties: the first hit of a rule is weighted at 100%, subsequent hits of the same rule are weighted at 30%, preventing score inflation from repeated matches of the same pattern.

## Evidence Index Mechanism

The system dynamically loads valid evidence filenames from `evidence_manifest.md` for citation fraud detection and fact source binding detection. Example manifest format:

## Audit Harness (verify_agent.py)

`scripts/verify_agent.py` implements the `LegalHarness` class with the following core methods:

| Method | Function | Detection Scope |
|:---|:---|:---|
| `load_and_compile()` | Load evidence index + complaint | Build valid evidence filename set, claim amount limits |
| `verify_structure()` | Document structure check | Four required section headings |
| `verify_citation_fraud()` | Citation fraud detection | Evidence citations vs. evidence index |
| `verify_strict_scope_containment()` | Claim boundary audit | Judgment amounts vs. claim limits |
| `verify_syllogism_complete_chain()` | Syllogism completeness audit | Citation anti-fraud + reasoning major/minor premise |
| `verify_fact_source_binding()` | Fact source binding detection | Each fact statement annotated with evidence source |
| `run_all_checks()` | Run all checks | Aggregate errors, output PASSED/FAILED |

**Iterative Fix Loop**: When the script returns `[AUDIT_FAILED]`, you must confirm with the user whether corrections are needed. If so, modify the document and re-run the script until it returns `[AUDIT_PASSED]`.

## FAQ

### General

**Q: Does this server call LLM APIs? Do I need an API key?**

A: No. The bridge architecture is designed to make zero LLM API calls. The rule engine returns results directly based on regex and numeric comparison. Semantic judgments leverage the AI Agent's own LLM; the server only provides prompt templates.

**Q: What types of legal documents are supported?**

A: Currently focused on Chinese labor dispute civil judgments. Future expansion to other civil/commercial cases is planned (see [Dev Plan](./docs/dev_plan_v2.md), OPT-20 multilingual support).

**Q: How fast is detection?**

A: Pure rule-engine detection is very fast (typically < 5 seconds) since it's based on regex matching and numeric comparison with no network requests. Vector retrieval or online verification may take slightly longer.

### Configuration

**Q: MCP client can't connect to the server?**

A: Check the following:
1. Python path is correct (`which python`)
2. `PYTHONPATH` includes the `src/` directory
3. Installed in editable mode (`pip install -e .`)
4. MCP client has trusted the server

**Q: Environment variables not taking effect?**

A: Ensure variable names use the `LH_` prefix (e.g., `LH_EVIDENCE_MANIFEST_PATH`), as specified in the `DetectionConfig` class in `config.py`.

### Detection

**Q: Why does my judgment pass audit but still have a hallucination score?**

A: The audit harness (verify_agent) only checks structure, citations, claim boundaries, and syllogism completeness. More granular detection (fact fabrication, law application, subjective rhetoric, etc.) requires running `run_rule_engine_full` or using semantic detection templates.

**Q: Evidence citation detection has too many false positives?**

A: Evidence citations must exactly match filenames in `evidence_manifest.md` (without extensions). Ensure manifest entries match judgment references. Short citations (≤4 characters) may have lower match rates — this is a known issue (OPT-04).

**Q: How to prevent inflated hallucination scores?**

A: The system already uses diminishing penalty weights. If scores still seem high, check whether the same pattern is being flagged repeatedly — this indicates the judgment draft itself needs fixing at the source rather than repeated detection.
